1. Abstract

在长上下文推理任务中，现有 Transformer 或标准状态空间模型通常难以同时兼顾 信息保真、推理深度与显存效率，尤其当上下文长度提升到 64k tokens 以上时，检索噪声与状态漂移会明显恶化最终表现。

为了解决这一问题，本文提出 Selective State Space Reader (S3R)， 通过选择性门控更新、低秩记忆压缩与证据片段回写机制，在保持长期依赖建模能力的同时， 将状态更新成本约束在可控范围内。

实验表明，该方法在长文问答、代码仓库导航与多跳文档推理任务上均取得更优表现，并显著降低了推理时显存开销。

模板建议 这一段很适合直接承接 paper-weaver 的三句式摘要输出。后续真实产物可以把每一句保留为独立段落， 让页面在“快速扫读”时更清晰。

2. Introduction / GAP

该方向的核心目标，是构建一种能够在超长上下文场景下仍然稳定保留关键信息、并支持复杂推理链的阅读模型。 现有方案大体可以分为三类：全量注意力模型、检索增强模型、状态压缩模型。

现有路线 A：全量注意力

优点是表达力强；缺点是显存与计算成本随上下文快速膨胀。

现有路线 B：检索增强

优点是速度快；缺点是证据召回与最终推理链之间容易断裂。

现有路线 C：状态压缩

优点是效率高；缺点是压缩后常牺牲细粒度可追踪性。

核心 GAP

现有方法往往无法同时满足 长程保真 与 证据可解释。

维度 现有问题 本文切入点
长上下文 长度提升时噪声累积严重 选择性状态更新
推理链 检索与答案生成断层 证据片段回写
效率 全量建模成本过高 低秩记忆压缩

3. Method Overview

方法主线可以概括为三步：先对输入序列生成可学习的重要性分数，再只对关键片段进行高保真状态写入， 最后将压缩状态与原始证据索引共同送入推理头。

  1. Evidence Scoring：给每个片段分配选择性分数。
  2. State Update：只对高价值片段执行完整状态更新。
  3. Reasoning Readout：输出答案时同时回溯对应证据片段。
一句话人话版 它不是“把所有东西都记住”，而是“只认真记值得记的内容，并保留为什么记它的证据索引”。

6. Code Appendix

这里用来验证代码块是不是走 GitHub 风格主题。后续 HTML 报告里如果需要插入伪代码、 关键实现片段或命令行示例，都可以复用这一段结构。 

import torch
import torch.nn as nn


class SelectiveStateBlock(nn.Module):
    def __init__(self, dim: int):
        super().__init__()
        self.gate = nn.Linear(dim, dim)
        self.proj = nn.Linear(dim, dim)

    def forward(self, x, state):
        gate = torch.sigmoid(self.gate(x))
        proposal = self.proj(x)
        next_state = (1 - gate) * state + gate * proposal
        return next_state
