# 变量速查表

读论文时对照此表抽取信息，整理成 `values.json` 后交给 `fill_lineage.py`。

## stage1（①起源段，必填 12 个字段）

| 变量 | 含义 |
|---|---|
| O | 起源驱动（理论/生物启发/物理建模） |
| X | 核心问题（如长距离依赖、动态建模） |
| Y | 模型框架类型 |
| A | 早期作者，尽量只写人名（如"Voelker"），不要在 A 里顺手把年份也写进去（比如"Voelker（2019）及后续工作"）——模板会再加一次{yA}的括号，变成年份重复两次。如果确实有多位作者/多篇后续工作要带出来，放进 mA 或 opt 的描述里说，不要塞进 A |
| yA | A 的发表年份 |
| mA | A 提出的方法（不可训练/固定结构/启发式） |
| opt | A 阶段的优化手段（如卷积/近似/离散化） |
| B | 扩展作者（同样建议只写人名，原因见上面 A 的说明） |
| yB | B 的发表年份 |
| ext | B 的扩展形式（连续时间/函数近似/结构化框架） |
| struct | B 引入的关键结构设计 |
| limA | 阶段①遗留限制（固定结构/不可学习参数/强假设）。不要让它以"依赖/受限于"这类词开头，模板里已经写了"仍依赖{limA}"，重复了会变成"仍依赖依赖…" |

## transitions（②③④...转折段，每段必填 6 个字段）

链条里每出现一位“把上一阶段的方法变得可训练/可扩展”的作者，就对应一个 transition。
第一个 transition 的 `prev_form` 由脚本自动接 stage1 的 `ext`；之后每个 transition 的
`prev_form` 自动接上一个 transition 的 `tform`，不需要手动填这一项。

| 变量 | 含义 |
|---|---|
| author | 该阶段作者，同样只写人名，原因见上面 stage1.A 的说明 |
| year | 发表年份 |
| tform | 提出的可训练/深层模型形式 |
| param | 被学习的核心参数（如状态矩阵 A） |
| cost | 新增代价（计算复杂度/内存/训练稳定性） |
| lim | 阶段遗留限制（扩展性/大规模应用受限）。模板里已经写了"导致在扩展性/大规模应用上仍存在{lim}"，所以这里直接写具体限制内容就行（比如"工程实现复杂度高"），不要再重复"扩展性受限"这种和前半句重叠的话 |

## final（总结段，必填 4 个字段）

| 变量 | 含义 |
|---|---|
| M | 本文方法 |
| design | 本文的参数化设计/算法结构 |
| gen | 本文推广到的更广模型类别（不要自带"更广泛的"前缀，模板里已经写了） |
| role | 本文在该脉络中的作用（改进/统一/推进） |

## 如何判断 transitions 该填几个

- 链条里有且只有 1 位作者做了“可训练化”这件事 → `--transitions 1`（标准三段式：①②③）
- 链条更长，比如 HiPPO（不可训练）→ LSSL（结构化扩展，仍算在①里）→ S4（可训练化，第 1 个 transition）
  → S5（简化参数化，仍是“让上一形式更好训练/更快”，算第 2 个 transition）
  → 本文方法引用到这里为止 → `--transitions 2`
- 不确定时，先数一数“链条里有几次‘让前一种方法变得可学习/更高效/更可扩展’的跃迁”，那就是 transitions 的数量。
