# 变量速查表

读 Introduction 时对照此表抽取信息，整理成 `values.json` 后交给 `fill_intro.py`。
结构固定为三段，不需要像 Lineage Loom 那样规划段落数。

## stage1（①研究领域+研究目标，4 个必填字段 + 1 个可选字段）

| 变量 | 含义 |
|---|---|
| X | 研究领域 |
| Y | （可选）核心问题/研究对象或现象。如果这和研究目标本质上是同一句话的两种说法，就不要在 values.json 里写这个 key——脚本会自动只渲染一句话，不会硬凑出两句重复内容 |
| goal_verb | 研究目标的动词（解释/预测/改善，三选一或近义词） |
| Z | 目标能力——本文最终要改进/提升的那个能力。**这个字段会在③里被原样重新提起**，用来检验故事线是否闭环：开头立的目标，结尾有没有真的兑现。不要让 Z 以"对"字开头，否则会撞上③模板里的"实现对…的改进" |
| A | 已有研究路径或方法集合 |

## stage2（②现有局限+研究缺口 GAP，必填 5 个字段，其中 limitations 是列表）

| 变量 | 含义 |
|---|---|
| B | 关键情境/条件/范围（如长序列、跨域迁移、高噪声数据） |
| limitations | 局限列表（数组），至少 1 项，建议 2-4 项；渲染时会自动用"、"连接。建议写成"某方法：具体问题"的形式（如"Transformer：计算复杂度过高"），比笼统的形容词更接近论文原文的对话感 |
| core_problem | GAP 所在的核心问题描述 |
| req1 | 现有方法无法同时满足的关键要求 1 |
| req2 | 现有方法无法同时满足的关键要求 2（req1 与 req2 的"无法兼得"就是 GAP 本身） |

## stage3（③本文工作+解决路径，必填 2 个字段 + 2 个可选字段）

| 变量 | 含义 |
|---|---|
| M | 本文提出的新方法/新理论/新框架/新模型 |
| mechanism | 关键机制/核心思想/方法路径 |
| old_structure | （可选，要和 new_form 一起填或一起不填）被重新表述或转化的原有问题结构。如果本文只是"通过机制直接提升效果"，没有显式的"把旧结构转化成新形式"这一步，就把这两个字段都从 values.json 里删掉 |
| new_form | （可选，规则同上）转化后的新的解释/分析/解决形式 |

注意：stage3 不需要再填 Z，脚本会自动复用 stage1 里的 Z，保证开头立的目标和结尾兑现的目标是同一个。

## 读 Introduction 的方法（对应"分"这一步）

不要把 Introduction 当成背景介绍来读，它的真正功能是制造"为什么必须有这篇论文"的理由。
读之前先把这篇论文放进它所在的研究对话里——它在跟哪几篇前作对话？是在回应谁、反驳谁、
接着谁往下做？想清楚这层关系，再去抽取变量，比直接摘抄背景句子更容易找到真正的 GAP。

## 做：GAP 表

render_intro.py 渲染完段落后会自动生成一张 GAP 表（B / 局限 / 核心问题 / req1 vs req2 /
本文方法 M / 核心机制 / 目标能力 Z），这张表就是"做"这一步要填的东西，可以直接当读论文
笔记的速记卡用。

## 判：能不能说出作者插进了哪条研究缝隙

render_intro.py 还会自动合成一句话："作者插入的研究缝隙是：现有方法无法同时满足『req1』
与『req2』；本文通过『mechanism』提出『M』，弥合了这一缝隙，使『Z』得以改进。"

如果这句话读起来含糊、循环论证，或者 Z 跟①里立的目标对不上，说明 req1/req2/mechanism
抽取得还不够具体——回头重新读 Introduction，而不是硬凑一句话糊弄过去。
