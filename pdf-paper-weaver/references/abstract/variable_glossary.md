# 变量速查表

读论文摘要时对照此表抽取信息，整理成 `values.json` 后交给 `fill_summary.py`。
结构固定为三句，不需要像 Lineage Loom 那样规划段落数。

## why（①问题+背景，Why，必填 4 个字段）

| 变量 | 含义 |
|---|---|
| task | 论文研究的具体任务/场景，不要带"任务"二字（模板里已经写了"在{task}任务中"） |
| methods | 现有方法（A/B/C），列出被比较的旧方法 |
| capability | 现有方法欠缺的关键能力 |
| scenario | 表现受限的具体场景（如长序列/高复杂度/大规模数据） |

## how（②方法+核心机制，How，必填 4 个字段）

| 变量 | 含义 |
|---|---|
| method_name | 本文提出的方法名称 |
| mechanism | 核心机制（关键技术/结构变换/优化方式） |
| capability_achieved | 该机制实现的能力 |
| effect | 由此带来的效果，写成完整短句（如"将计算复杂度降低到接近线性"或"大幅提升长程依赖建模的性能"），不要再加"从而"（模板已经写了） |

## so_what（③结果+贡献，So what，必填 2 个字段）

| 变量 | 含义 |
|---|---|
| benchmark | 实验验证用的数据集/基准 |
| improvement | 取得的具体改进，写成名词短语（如"多项长序列任务上的最优表现"），不要再加"能够实现"（模板已经写了） |

## 读摘要的顺序（对应"分"这一步）

不要先追公式里的数学符号。先在心里（或笔记里）按顺序回答三个问题，再去抽取变量：

1. 它研究什么？→ 对应 why.task
2. 旧方法哪里不足？→ 对应 why.methods / why.capability / why.scenario
3. 它提出什么？→ 对应 how 的四个字段，以及 so_what 的两个字段

这个顺序本身就是"总-分-做-判"里"分"的内容：先理解，再填空，比直接对着摘要原文抠词翻译更不容易跑偏。

## 判定自检（render_summary.py 自动跑）

渲染后脚本会检查三句话是否各自命中至少一个角色关键词：

- 第①句应出现"不足/受限/局限"之一（问题）
- 第②句应出现"提出"且出现"通过"（方法）
- 第③句应出现"实验表明"等证据类词（证据）

如果某一句没有命中，说明对应字段写得太空泛或者被改写掉了关键词，回头检查一下。
